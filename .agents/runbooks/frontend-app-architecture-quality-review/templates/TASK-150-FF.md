# Feature Flag / Deliver Flag / Config / 开关

> updated_by: Kilo - GLM-5
> updated_at: 2026-06-01 12:08:20

## 审查目标

- 判断前端应用是否具备统一的 Feature Flag / Deliver Flag / Config / 开关基础设施，明确各类控制机制的语义边界与适用场景。
- 审查 Flag/Config 的全生命周期管理：定义、获取、缓存、评估、降级、过期与清理，确保链路完整可控。
- 识别 Flag/Config 对应用启动时序、业务流程、安全合规与可观测性的影响，评估架构层面的风险与改进机会。
- 确保远程配置与本地开关之间的优先级、覆盖与回退策略清晰，避免因配置不一致导致的线上事故。

## 核心关注点

- **分类与语义边界**：
  - Feature Flag（功能开关）：控制功能模块的可见性与可用性，是否按布尔/多值/百分比等类型明确分类
  - Deliver Flag（投放开关）：控制灰度投放、分桶策略、A/B 实验分组，是否与 Feature Flag 在概念与实现上清晰区分
  - Config（远程配置）：服务端下发的键值对配置项（业务参数、阈值、URL 等），是否与开关类机制在数据模型与使用方式上有明确边界
  - 本地开关（硬编码 / BuildConfig / 本地文件）：是否作为远程配置的兜底或开发阶段便利手段，是否在上线后遗留无用开关
  - 上述四类是否在代码中有统一的抽象模型，还是散落在各模块以不同方式实现

- **统一抽象与 API 设计**：
  - 是否存在统一的 Flag/Config 读取 API（如 `FlagProvider.getBoolean(key)`、`ConfigProvider.getString(key, defaultValue)`）
  - API 是否类型安全（避免纯字符串 key + Any 类型的弱类型访问），是否支持枚举/密封类约束
  - Flag 评估逻辑是否集中（如 Provider 内部处理百分比、分桶、受众规则），还是散落在业务代码中
  - 是否支持 Flag 的命名空间或分组（如按模块/业务域前缀），避免 key 冲突与无序膨胀

- **远程获取与缓存策略**：
  - 远程 Flag/Config 的获取时机：是否在应用启动阶段同步阻塞获取，是否支持异步/懒加载，是否影响首屏渲染
  - 缓存策略：是否在本地持久化（SharedPreferences / Room / 文件等），缓存有效期与刷新策略是否明确
  - 缓存失效与回退：远程获取失败时是否使用上次缓存值或硬编码默认值，默认值是否集中管理
  - 增量更新 vs 全量拉取：是否支持按需拉取变更配置，还是每次全量覆盖
  - 实时性要求：是否需要 WebSocket / 长轮询推送配置变更，还是仅靠定期拉取

- **灰度与分桶**：
  - 灰度策略是否由服务端决策（推荐），还是客户端本地计算分桶比例
  - 分桶依据（userId / deviceId / 随机数 / 地区 / 版本等）是否明确且稳定
  - 同一用户跨会话/跨设备的分桶一致性是否保证
  - A/B 实验分组与 Feature Flag 的关系是否清晰：实验是否基于 Flag 实现，实验结束后 Flag 是否及时清理

- **时序与初始化**：
  - Flag/Config SDK 的初始化时序是否与 Application.onCreate 中的其他 SDK 有明确的依赖顺序
  - 是否存在在 Flag 初始化完成前就读取 Flag 值的场景，此时返回值是否为安全的默认值
  - 首次安装（无缓存）与升级（有旧缓存但可能过期）场景下的 Flag 评估行为是否明确
  - Flag 变更后的实时生效范围：是否需要重启、是否仅新页面生效、是否影响已打开页面

- **默认值与降级**：
  - 每个 Flag/Config 是否都有显式声明的默认值，默认值是否合理（宁可关闭功能而非开启半成品）
  - 默认值是否集中管理（如常量类/枚举），还是散落在各调用点
  - 远程服务不可用、网络异常、响应超时、响应格式异常等场景的降级策略是否覆盖
  - 降级路径是否经过测试验证，是否可能出现因降级导致的空指针或类型转换异常

- **安全与合规**：
  - Flag/Config 中是否包含敏感信息（API Key / 密钥 / 内部 URL / 调试开关），是否可能通过明文缓存或日志泄露
  - 远程配置通道是否加密（HTTPS + 报文加密），是否验证配置完整性（签名校验）
  - 客户端是否可被篡改 Flag 值（如通过 Hook / 修改本地缓存），关键业务逻辑是否仅依赖服务端校验
  - 调试/内部开关在生产环境是否自动关闭，是否存在开发便利开关泄露到线上的风险

- **生命周期管理**：
  - Flag/Config 的创建、使用、废弃是否有完整流程：是否有注册/审批机制，废弃后是否及时清理代码引用
  - 是否存在大量僵尸开关（已永久开启/关闭但代码仍保留），僵尸开关是否影响代码可读性与包体
  - 临时开关（如紧急修复、活动开关）是否有自动过期机制（TTL），过期后是否自动降级到默认值
  - 是否有 Flag/Config 的全量清单与文档，是否可追溯每个 Flag 的用途、负责人与状态

- **可观测性**：
  - Flag 评估结果是否可追踪：是否在日志/埋点中记录 key、值、来源（远程/缓存/默认）、耗时
  - 远程配置获取是否可监控：成功率、延迟、缓存命中率、降级率是否有指标与告警
  - 是否支持远程调试：是否可在不发版的情况下调整某个用户的 Flag 值以排查问题
  - Flag 变更是否可审计：是否记录谁在何时修改了什么 Flag，变更是否可回滚

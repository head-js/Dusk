# Utils（工具层）

> updated_by: Cascade - Cascade
> updated_at: 2026-05-06 11:40:00

## 审查目标

- 将通用能力从业务中抽离为工具层，提升代码复用率，减少重复实现。
- 控制工具类的职责范围，避免形成难以测试和理解的"工具黑洞"。
- 确保工具层对框架运行时（响应式、生命周期、全局单例等）的依赖是清晰、可控且不会引入内存泄漏。

## 核心关注点

- **本地存储与缓存（Storage）**：
  - 存储分层是否清晰：KV（配置/状态）vs 文件缓存（可过期）vs DB（结构化/查询）
  - 是否避免在工具层散落多套存储实现（多套 SP/MMKV/DB/文件缓存并存）且缺少治理规范
  - Key/Schema 是否集中治理：命名规范、命名空间/模块隔离、迁移策略（升级/灰度/回滚）
  - 数据生命周期是否可控：登录/登出、切换账号、撤回同意、风控触发等场景下的清理策略是否明确
  - 安全与合规：敏感数据是否最小化落盘；必须落盘时是否加密；备份/导出/Root 风险是否评估
  - 性能与可靠性：主线程写入控制（commit/apply）、高频写入合并、容量上限/淘汰策略、读写失败兜底
  - 可观测性：命中率、占用、清理耗时、写入失败率等指标是否可监控
  - **审查报告**：`/.plans/架构质量审查-310-Utils-Storage.md`

- **网络与请求（Http）**：
  - 是否有统一的请求客户端与分层封装（base client / service / api modules），避免页面直调
  - 请求入参与响应的类型/Schema 是否有约束（TS 类型、运行时校验、转换层），避免隐式字段漂移
  - 是否有统一的超时、重试、取消（AbortController）、并发控制与防抖/合并策略
  - 错误处理是否标准化（错误码映射、业务错误 vs 系统错误、用户提示口径）
  - 是否支持可观测性（requestId、链路追踪、耗时统计、失败原因分桶）
  - 安全与合规（敏感 header/参数脱敏、证书/HTTPS、token 注入边界）
  - **审查报告**：`/.plans/架构质量审查-320-Utils-Http.md`

- **WebSocket**：
  - 连接管理是否集中（单例/多实例策略、页面生命周期绑定、资源释放）
  - 心跳、重连、退避与网络切换策略是否可控（前后台、弱网、断网恢复）
  - 消息协议是否版本化与可验证（Schema、ack/seq、幂等与乱序处理）
  - 安全与鉴权是否明确（token 刷新、过期处理、订阅权限边界）
  - 可观测性是否完善（连接状态、收发量、失败原因、延迟与告警）
  - **审查报告**：`/.plans/架构质量审查-325-Utils-WebSocket.md`

- **日志（Log）**：
  - 日志分级与环境策略是否清晰（dev/test/prod 的输出与采样）
  - 是否有统一的日志 API 与结构化字段（模块、事件名、traceId/requestId、用户态）
  - 是否避免敏感信息泄露（脱敏、黑名单字段、PII 最小化）
  - 是否支持持久化/上传与失败兜底（离线缓存、重试、限流）
  - 是否与错误收集/性能指标联动形成闭环（异常、关键动作、请求）
  - **审查报告**：`/.plans/架构质量审查-330-Utils-Log.md`

- **设备信息（Device Info）**：
  - 设备信息采集是否最小化（仅采集业务必需字段），避免过度采集硬件标识与可识别信息
  - 字段来源与口径是否统一（系统 API、SDK 回传、自定义拼装），避免同一指标多源漂移
  - 设备指纹相关逻辑是否具备合规约束（告知/同意/撤回后降级），并避免将敏感标识明文落盘
  - 采集失败、权限受限、系统版本差异场景是否有兜底与降级策略
  - 是否建立可观测性（采集成功率、字段缺失率、上报延迟、异常告警）
  - **审查报告**：`/.plans/架构质量审查-335-Utils-设备信息.md`

- **加解密**：
  - 算法与实现是否统一入口（加密/解密/签名/验签），避免散落多套实现
  - Key/IV/盐值等敏感材料的管理是否合规（不硬编码、生命周期、环境隔离）
  - 编码与格式是否统一（base64/hex、padding、字符集），避免跨端不一致
  - 异常与兼容策略是否明确（旧算法迁移、失败降级、错误可观测）
  - 性能与安全权衡是否可控（大对象处理、流式、是否误用弱算法）
  - **审查报告**：`/.plans/架构质量审查-340-Utils-加解密.md`

- **日期时间与时区（DateTime / Timezone）**：
  - 日期解析与格式化是否统一入口（避免各处硬编码 `yyyy-MM-dd HH:mm:ss`、`dd/MM/yyyy`、ISO 模板）
  - 时区与 Locale 策略是否清晰（展示使用本地时区、服务端交互使用约定时区），并避免隐式依赖系统默认值
  - 时间戳单位是否显式治理（`timestampMs` / `timestampSec`），避免秒/毫秒混用导致比较与排序错误
  - 是否覆盖夏令时、跨时区、跨日边界（00:00 附近）等关键边界场景，避免日期偏移与展示错误
  - 国家/地区日期展示规则是否配置化，避免按国家码散落条件分支造成维护成本上升
  - **审查报告**：`/.plans/架构质量审查-360-Utils-DateTime.md`

- **Currency**：
  - 货币金额的表示是否统一（整数最小单位 vs Decimal），避免浮点误差
  - 格式化与舍入规则是否集中治理（round/floor/ceil、银行家舍入、展示精度）
  - 多币种与汇率来源是否明确（精度、时效、缓存、失败兜底）
  - i18n 与本地化展示是否一致（符号位置、千分位、小数点、负数）
  - **审查报告**：`/.plans/架构质量审查-365-Utils-Currency.md`

- **Icon**：
  - 图标资源的组织与命名是否规范（命名空间、分模块、去重）
  - 加载策略是否合理（按需加载、tree-shaking、sprite/inline 选择）
  - 主题与多态支持是否清晰（颜色/尺寸/暗色模式、状态态图）
  - 性能与包体影响是否可控（重复资源、无用图标清理、缓存）
  - 可访问性是否考虑（aria-label、对比度、可替代文本策略）
  - **审查报告**：`/.plans/架构质量审查-367-Utils-Icon.md`

- **报表**：
  - 指标口径与数据源是否明确（字段定义、过滤条件、时间范围、维度）
  - 查询与聚合性能是否可控（分页、缓存、异步、限流、导出策略）
  - 展示与导出格式是否统一（日期/货币/百分比格式、排序口径、精度）
  - 权限与审计是否完善（数据权限、导出审批/水印、操作日志）
  - 异常与边界处理是否明确（空数据、超时、部分失败、重试与提示）
  - **审查报告**：`/.plans/架构质量审查-390-Utils-报表.md`

## 其它关注点

- **职责单一性**：
  - 单个工具类/方法是否聚焦单一领域（如日期、字符串、文件、网络）
  - 是否存在"大杂烩式"工具类
  - 复用逻辑单元（如 mixins/composables/hooks/HOC 等）是否职责单一、可复用
  - 是否存在“工具层夹带业务规则/数据结构”的情况，导致工具层难以复用与迁移

- **命名与组织**：
  - 文件结构与命名是否清晰反映用途
  - 是否存在大量"CommonUtils"、"Utils2"等模糊命名
  - 复用逻辑单元命名是否清晰描述功能域
  - 工具函数是否存在重复实现（格式化/解析/下载/加密/日志等能力散落在页面/组件）

- **静态方法与全局状态**：
  - 是否过度使用全局状态，导致难以管理
  - 是否正确使用当前项目的状态管理方案管理全局状态
  - 是否避免在工具函数中直接操作 DOM

- **响应式使用**：
  - 工具层是否不依赖具体 UI 运行时/组件实例（除非通过明确的适配层注入）
  - 是否避免长期持有组件引用或订阅句柄，导致内存泄漏
  - 回调与上下文绑定是否可控（事件/定时器/订阅的注册与释放是否成对）

- **平台依赖与资源句柄**：
  - 对系统上下文/权限/文件句柄/网络连接等依赖是否通过明确边界注入与治理，避免隐式持有与跨层泄漏
  - 资源释放是否有统一约束（超时、取消、close/dispose），并可观测

- **可测试性**：
  - 工具方法是否易于通过单元测试验证
  - 是否与 UI 或外部环境强耦合导致无法隔离测试
  - 复用逻辑单元（如 mixins/composables/hooks/HOC 等）是否易于测试
  - 关键工具函数是否覆盖边界条件（空值/异常输入/编码/时区等）并具备回归保障

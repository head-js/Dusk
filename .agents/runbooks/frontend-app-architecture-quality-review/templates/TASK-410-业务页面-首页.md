# 首页

> updated_by: Kilo - GLM-5
> updated_at: 2026-06-02 20:03:00

## 审查意图

首页是用户启动 App 后的第一个着陆点，也是绝大多数业务流的起点。本审查的核心目标是**以用户视角，从启动到首屏再到一级页面，梳理完整的页面流向与登录守卫机制**，识别流程断点与守卫缺口，而非对单个页面做技术优化分析。

## 审查范围与方向

### 1. 启动链路
- 从 Launcher Activity 到首页容器的完整跳转链路
- 启动过渡期间的准备动作（登录态检查、配置拉取、广告 ID 获取等）是否合理
- 延迟策略是否可优化（硬编码延迟 vs 数据就绪即跳转）
- 是否区分登录/未登录用户的启动体验

### 2. 首屏状态机
- 首页是否根据用户生命周期阶段展示不同内容（如未登录 / 已登录未开通 / 流程进行中 / 已完成关键动作等）
- 状态由前端判定还是后端驱动（如 pageFlag），驱动方式是否足够纯粹
- 各状态之间的流转是否完整、有无遗漏分支
- 新旧状态逻辑是否并存，是否存在未清理的遗留代码

### 3. 首页典型页面流向
- 从首页出发的核心业务跳转路径（主流程入口、次要功能入口、消息/通知等）
- 多步骤流程的节点路由机制：是否由统一的路由调度器管理，节点覆盖是否完整
- 首页操作触发的深层流程是否能正确回退或回跳

### 4. 登录守卫
- 是否存在统一的路由拦截器，在跳转需登录页面时自动拦截
- 若无统一守卫：Token 失效时后端返回什么错误码，前端如何处理，跳转目标是登录页还是首页
- 各业务页面手动检查登录态的覆盖度，是否有遗漏
- 登录成功后是否能回到用户原本想访问的页面（回跳机制）

### 5. 首页相关一级页面
- 登录页的结构与流程（Activity 壳 + 内部 Fragment 导航 vs 独立页面链）
- 其他一级 Tab（如优惠券、个人中心）与首页的关系，仅需概述，不深入 drill

## 不覆盖

本审查不覆盖纯技术优化项（骨架屏、DiffUtil、嵌套滑动、ViewModel 迁移等），这些由独立技术审查任务负责。

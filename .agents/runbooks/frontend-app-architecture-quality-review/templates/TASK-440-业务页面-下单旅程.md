# 下单旅程

> updated_by: Kilo - GLM-5
> updated_at: 2026-06-03 14:58:00

## 审查目标

审查业务下单旅程中 Page 间的流转质量，确保用户从入口到终态的每一步导航都是正确、可恢复、不卡死的。

## 审查粒度

**以 Page 为最小审查单位**。Page 在前端项目中通常体现为：

| 载体 | 识别方式 | 备注 |
|------|----------|------|
| Activity / ViewController | 平台标准页面 | 主要载体 |
| Fragment / 子路由 | 可替换页面片段 | 当它充当独立页面时 |
| 自定义 View / Component | 当它承担完整页面职责时 | 需辨析：是 Page 还是 Page 内组件 |

**判断标准**：如果一个 View 具有独立的数据加载、用户交互、提交/跳转逻辑，且用户感知到它是一个"页面"，则视为 Page；如果它只是 Page 内的展示组件（如卡片、输入框组），则不是。

## 审查边界

### ✅ 属于旅程审查

| 方向 | 关注点 |
|------|--------|
| **Page → Page 导航** | 导航路径是否清晰、是否存在死路 |
| **导航触发条件** | 什么条件下跳转到下一页（后端驱动、用户状态判断、按钮点击） |
| **Submit 后流向** | 提交成功/失败后用户被导航到哪里 |
| **页面间数据传递** | Page 间传递了什么数据、数据是否一致 |
| **一致性映射** | 同一业务状态在不同 Page 中的映射是否一致 |
| **缓存数据对流转的影响** | Page 使用过期缓存数据导致错误导航决策 |
| **页面意外关闭** | 非预期的页面关闭/跳转（如事件总线残留事件） |
| **页面卡死** | Submit 失败后页面无响应，用户无出路 |
| **生命周期对页面状态的影响** | 页面恢复/返回后数据是否有效 |
| **断点续做** | 用户中途退出后能否恢复到断点 |
| **终态处理** | 流程最后一步是否正确闭环 |

### ❌ 不属于旅程审查（属于代码质量/安全审查）

| 方向 | 归属 | 原因 |
|------|------|------|
| 表单字段校验（必填、格式、正则） | 代码质量审查 | Page 内实现，不影响 Page 间流转 |
| 按钮防抖/门控机制 | 代码质量审查 | 实现细节，不影响流转 |
| 幂等 token 实现 | 代码质量审查 | 接口契约问题，不影响页面流向 |
| 加密实现细节 | 代码/安全审查 | Page 内安全措施，不影响流转 |
| 安全键盘/防截屏 | 代码/安全审查 | Page 内安全措施，不影响流转 |
| 变量命名/typo | 代码质量审查 | 不影响页面流转 |
| 日志脱敏 | 代码/安全审查 | 不影响页面流转 |
| Loading 状态管理 | 代码质量审查 | UI 状态管理问题 |

### ⚠️ 灰色地带（需结合上下文判断）

| 灰色地带 | 判断依据 | 归属 |
|----------|----------|------|
| 基类体系差异 | 如果影响页面数据刷新行为 → 旅程审查；如果仅是代码架构问题 → 代码架构审查 | 看影响 |
| 浮点精度 | 如果导致页面显示金额与实际不一致 → 旅程审查；如果是内部计算误差 → 代码质量审查 | 看影响 |
| 提交防重 | 如果导致重复进入成功页 → 旅程审查；如果是接口幂等实现 → 代码质量审查 | 看影响 |

**判断原则**：是否影响 **Page 间的流转** 或 **Page 对用户呈现的状态**？是 → 旅程审查；否 → 代码质量审查。

## 审查维度

| 维度 | 关注点 |
|------|--------|
| 路由层设计 | 入口页面的状态分发是否清晰、降级策略是否合理、新旧路由是否有残留 |
| 状态机设计 | 后端驱动的流程是否灵活可控、客户端是否覆盖所有状态、终态处理是否正确 |
| 导航架构 | 导航模式是否统一（声明式 vs 显式 vs 单例包装）、是否存在硬耦合 |
| 页面间数据一致性 | 跨 Page 传递的业务参数映射是否一致、缓存数据是否过期 |
| 生命周期与页面状态 | 页面恢复/返回时数据刷新策略、系统回收后状态恢复、页面断点 |
| Submit 后流向保障 | 成功/失败/异常后的用户出路、页面是否卡死 |
| 新旧 Page 共存 | 新旧版本页面并存对流转的影响、不同流程路径是否统一 |

## 审查步骤

1. **绘制状态机**：从入口页面出发，画出所有 Page 的导航路径
2. **识别导航触发条件**：每个跳转由什么条件驱动（后端指令、用户状态、用户操作）
3. **追踪数据传递**：每个跳转携带什么数据，数据在下一页如何消费
4. **验证一致性映射**：同一业务状态在不同 Page 中是否映射到相同值
5. **检查缓存依赖**：Page 是否依赖过期缓存做出导航决策
6. **测试 Submit 失败路径**：每个提交操作的失败回调是否给了用户出路
7. **验证终态闭环**：流程最后一步是否正确回到入口或进入结果页
8. **评估生命周期影响**：页面恢复/返回后页面状态是否可用
9. **识别新旧共存**：是否存在同一业务走不同版本 Page 的情况

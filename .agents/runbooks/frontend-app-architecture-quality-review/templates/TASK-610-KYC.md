# KYC（用户归一化与渠道归因）

> updated_by: Kilo - GLM-5
> updated_at: 2026-05-20 17:26:09

- **KYC（用户归一化与渠道归因）**：
  - 用户归一化：
    - 同一用户在不同登录方式（手机号/邮箱/第三方 OAuth/一键登录）之间是否可被识别与合并，归一化规则是否由服务端裁决、客户端仅做状态同步
    - 设备更换、重装、多设备登录等场景下，用户身份是否可被正确关联，是否存在"幽灵账号"或重复身份
    - 用户合并/解绑/换绑的触发时机与数据迁移策略是否明确，合并后历史数据（订单、授信、还款等）是否完整关联
    - 归一化标识（如 userId/merchantId/统一 ID）是否在全局一致使用，是否存在业务模块绕过统一标识直接使用局部 ID
    - 登录态与归一化状态的关系是否清晰：登录后是否必然触发身份归一化检查，归一化失败是否阻断关键流程
  - 渠道归因（channel / source / utm_x）：
    - channel / source / utm_source / utm_medium / utm_campaign / utm_content / utm_term 等归因参数的采集入口是否统一（如 DeepLink、Install Referrer、WebView 拦截、Push Payload），是否避免多入口重复采集与覆盖冲突
    - 归因参数的持久化策略是否明确：存储位置、生命周期、跨会话/跨版本保留策略，是否与用户归一化联动（如新用户注册时绑定首次归因）
    - 归因参数在业务链路中的传递是否完整：注册、授信、下单、放款等关键转化节点是否携带归因参数，是否与广告归因 SDK（如 AppsFlyer/Adjust）的口径对齐
    - 归因参数的优先级与覆盖规则是否清晰：多来源归因冲突时（如 DeepLink 与 Install Referrer 不一致）以谁为准，是否支持服务端下发覆盖
    - 归因参数的安全与合规：是否最小化采集，是否在隐私政策中披露，是否支持用户撤回后清理
    - 可观测性是否完善：归因参数缺失/异常/覆盖是否可追踪，归因链路端到端是否可审计
